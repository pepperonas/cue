# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`cue` тАФ a multi-tenant prompt-/todo-queue web app for managing Claude-Code-CLI prompts.
Capture prompts, group by project, work them through a status workflow (Queued тЖТ
Running тЖТ Done, plus Failed/Archived), and copy them into the CLI with one click.
Deployed at `https://cue.celox.io` behind a reverse proxy. Repo: `pepperonas/cue` (private).

Hard constraints: **multi-tenant** тАФ sign in with **Google OAuth**, each user owns their own
projects/prompts (`User` table + `user_id` FK on every owned row), access gated by an
email/domain allowlist. MD3 Expressive design with real spring motion, footer
`┬й 2026 Martin Pfeffer | celox.io` on every page.

## Commands

```bash
# Backend (from backend/)
uv venv && uv pip install -e ".[dev]"     # setup
uv run uvicorn app.main:app --reload --port 8000   # dev server (set CUE_DEV=1 COOKIE_SECURE=false)
uv run pytest                              # run all tests
uv run pytest tests/test_api.py::test_full_flow    # single test

# Frontend (from frontend/)
pnpm install
pnpm dev                                   # Vite dev server :5173, proxies /api -> :8000
pnpm build                                 # tsc -b && vite build -> dist/
pnpm typecheck

# Full app (prod): one container serves API + built frontend
docker compose up -d --build               # listens on 127.0.0.1:8791
# Auth needs GOOGLE_CLIENT_ID/SECRET + an allowlist in .env (see .env.example).
```

After backend or frontend changes, run `uv run pytest` and `pnpm build` before committing.

## Architecture

**One process, one port.** FastAPI mounts the JSON API as a sub-app at `/api` and serves
the built frontend (`STATIC_DIR`, default `static/`) as a SPA for all other paths, with an
`index.html` fallback so client-side state survives hard reloads. In dev the two run
separately and Vite proxies `/api` to `:8000`.

**Backend (`backend/app/`)**
- `main.py` тАФ app assembly, lifespan (`init_db`), security-header + CSP middleware, `/api` mount, SPA static serving (with path-traversal guard).
- `config.py` тАФ env-driven `Settings` (cached). `validate()` fails fast (needs `SECRET_KEY` + `GOOGLE_CLIENT_ID`/`SECRET`) unless `CUE_DEV=1`. `is_email_allowed()` enforces the allowlist (empty lists = closed in prod, open in dev). `google_redirect_uri` derives from `ALLOWED_ORIGIN`.
- `db.py` тАФ SQLite engine; a `connect` event sets `WAL`, `foreign_keys=ON`, `busy_timeout`. `init_db` runs `_migrate()` тАФ idempotent `ALTER TABLE` (PRAGMA-guarded) for additive columns (`bookmarked`, `bookmark_order`, and `user_id` on `prompt`/`project`), since `create_all` only adds missing *tables*, never columns (no Alembic).
- `models.py` тАФ `User` + `Project` + `Prompt` (SQLModel tables). `PromptStatus` enum. Every owned row has a `user_id` FK to `user`; `Prompt.bookmarked`/`bookmark_order` back the bookmarks section; `Prompt.tested` flags whether a running/done prompt's feature was tested. Project name uniqueness is **per user**, not global.
- `schemas.py` тАФ Pydantic request/response models (kept separate from table models). `MeResponse` carries `UserRead`.
- `security.py` тАФ itsdangerous signers: session tokens (payload carries `uid` + the CSRF secret), CSRF double-submit check, and short-lived OAuth-`state` tokens (`issue_oauth_state`/`oauth_state_valid`) to guard the Google redirect. No passwords.
- `deps.py` тАФ `current_session` (401 guard), `current_user_id` (extracts `uid` from the session тАФ inject this into every data router to scope by tenant), `require_csrf` (403 guard + Origin check), `get_client_ip`.
- `routers/` тАФ `auth` (Google OAuth), `projects`, `prompts`, `attachments`, `runs`, `importexport`. **Every data query is filtered by `user_id`; ownership is re-checked on get/update/delete (404 if not owned).**
- **Prompt capture** (`models.CaptureSession/CapturedPrompt`, `routers/capture.py`): every prompt typed in the Claude Code CLI is logged into cue via a `UserPromptSubmit` hook тЖТ spool тЖТ the cue-runner's forwarder тЖТ `POST /api/capture` (Bearer `CAPTURE_TOKEN`, attributed to `OWNER_EMAIL`'s user). Items carry `{session_id, cwd, git_root, prompt, seq}`; the project is derived under `CAPTURE_BASE` (`Settings.capture_project_name`, auto-creates the project): preferred from the hook-reported **git root** relative to the base with `_`-prefixed grouping folders skipped (`_customers/celox/website` тЖТ project `celox/website`; an all-underscore root keeps its own name), fallback = first non-`_` `cwd` segment (old hook / no repo / root outside base; `_customers/celox` тЖТ `celox`), grouped into a `CaptureSession` per Claude `session_id`, deduped on `(session, seq)`. Owner-facing history: `GET /api/sessions(/{id})` (session detail lists prompts newest-first, `seq` desc), `POST /api/sessions/{id}/prompts/{cp}/promote` (captured тЖТ real queued prompt), `DELETE /api/sessions/{id}`. **Multi-tenant:** `/api/capture` resolves the Bearer token to a user тАФ the env `CAPTURE_TOKEN` тЖТ `OWNER_EMAIL`, or a per-user `User.capture_token` тЖТ that user (with `User.project_base` for derivation). Sessions endpoints + the **Verlauf** tab are per-user (`current_user_id`, not owner-gated); `Settings тЖТ Prompt-Capture` lets any user set their base + (re)generate their token. The runner sets `CUE_NO_CAPTURE=1` on its own `claude` execs; its forwarder throttles retry errors. Hook lives in `~/.claude/settings.json` (`python3 тАж/cue-runner/hooks/capture_hook.py`); runner needs `CAPTURE_TOKEN` in `cue-runner/.env`. All phases done.
- **Send a prompt into a live CLI session** (`models.CliDelivery`, `routers/capture.py`, reverse of capture): the capture hook additionally records each session's terminal context (`ITERM_SESSION_ID`, `TMUX`/`TMUX_PANE` тЖТ `CaptureSession.term_program/iterm_session_id/tmux_pane/tmux_socket`, refreshed on every ingest); a session is `deliverable` when a transport is known. Owner-only `POST /api/sessions/{id}/send {text, submit}` enqueues a `CliDelivery` (`queued`); the runner claims it atomically via `GET /api/cli/claim` (guarded by `RUNNER_TOKEN`, mark `sending`), performs it, and reports via `POST /api/cli/{id}/result`; the frontend polls `GET /api/cli/{id}`. **Owner-only** (`require_owner`) because it drives a terminal on the runner's machine, exactly like runs. Runner transport = `cue_runner/deliver.py`: **iTerm2** via `osascript` (AppleScript `write text`, GUID from `ITERM_SESSION_ID`) and **tmux** via `paste-buffer`, both using **bracketed paste** (ESC[200~тАжESC[201~) so multi-line prompts land as literal input; then an optional Enter to submit. All ids validated + argv-only (no shell/AppleScript injection). Runner delivery loop polls `GET /api/cli/claim` every `DELIVER_INTERVAL` (1.5 s; `CUE_DELIVER=0` disables). **iTerm2 needs a one-time TCC Automation grant** for the runner's process. Frontend: `SendToSessionDialog` + "An CLI senden" in `DetailSheet` (gated on `canRun`).
- **Run engine** (`models.Run/RunStep/RunLog`, `routers/runs.py`): execute saved prompts through the Claude Code CLI on a Mac **runner** that polls cue. A `Run` is `single` (one step) or `chain` (ordered steps in one Claude session); `RunStep` snapshots the prompt text (reproducible). **User endpoints** (`POST/GET /runs`, `GET /runs/{id}?after_seq=`, `POST /runs/{id}/cancel`, `GET /runs/config`) use cookie auth and are **owner-only** (`require_owner` тЖТ `OWNER_EMAIL`; the run executes code on the runner's machine, so it must not be open to other allowlisted users). **Runner endpoints** (`POST /runs/claim`, `/runs/{id}/heartbeat|log`, `/runs/{id}/steps/{idx}/result`, `/runs/{id}/result`) are guarded only by `RUNNER_TOKEN` (`require_runner`, Bearer). Claim is **atomic** (single `UPDATE ... WHERE status='queued' ... RETURNING id`). `project_path` is whitelisted via `Settings.is_path_allowed()` against `ALLOWED_PROJECT_BASES` (string-only on the VPS; the runner re-validates). Chains thread one Claude session; the runner pre-assigns it via `--session-id <uuid>` then `--resume` (CLI v2.1.195 has **no `--cwd`** тЖТ set subprocess cwd; **no `--max-turns`** тЖТ rely on per-step timeout + `--permission-mode`). Cancel sets `cancel_requested` (running) or `canceled` (queued); the runner learns it from the heartbeat response. The Mac daemon lives in **`cue-runner/`** (own venv/pyproject, PM2-able): asyncio claim-loop, step exec via `create_subprocess_exec` (sets `cwd`, `stdin=DEVNULL` to skip the CLI's stdin wait, parses `stream-json`), heartbeat (also picks up cancel), SIGTERMтЖТSIGKILL cancel, per-step timeout; its own `ALLOWED_BASES` re-validation. `cue-runner/` is **not** deployed to the VPS тАФ it runs on the Mac where `claude` is authenticated. **Frontend** (`RunsView`, `RunDialog`, `state/queries` `useRunConfig/useRuns/useRun/useCreateRun/useCancelRun`): the **Runs** tab + the prompt "Ausf├╝hren" action + select-bar "Ausf├╝hren" (1 prompt = single, тЙе2 = chain) are gated on `useRunConfig` succeeding (owner-only; a 403 hides them, `canRun`). `RunDialog` picks a project base from the whitelist + optional subfolder, model/permission-mode/allowed-tools/bare/skip-permissions/stop-on-error; `RunsView` polls (`refetchInterval`) for a live status + log tail, copyable session id, cost, cancel, and re-run. **Hardening (P4):** `runs.reap_stale()` fails runs stuck in claiming/running whose `last_heartbeat` is older than `RUN_STALE_TIMEOUT` (also cleans up runs left in-flight across a backend restart); a `_run_reaper_loop` in the lifespan runs it at startup and every 60 s. The runner heart-beats independently of step progress, so long steps aren't falsely reaped. (P1 backend + P2 runner + P3 frontend + P4 hardening done; P5 = e2e/docs/version + live runner.)
- **Attachments (screenshots)**: `Attachment` table (`user_id`, nullable `prompt_id`, stored `filename`, `original_name`, `content_type`, `size`). Files live in `ATTACHMENTS_DIR` (set to `/data/attachments` in prod so they persist on the volume). `routers/attachments.py`: `POST /attachments` (multipart, image-only, тЙд`MAX_ATTACHMENT_BYTES`) returns `{id,url,name,...}` with `prompt_id=null` (pending); `GET /attachments/{id}` serves the file (ownership-checked, not static); `DELETE /attachments/{id}`. Prompts associate them via `attachment_ids` on create/update (`_attach`), and `PromptRead.attachments` is populated by `_reads()`/`_read()` in `prompts.py` (batch-loaded). `delete_prompt` purges files; `merge` reassigns source screenshots to the merged prompt when originals are deleted. **Auto-expiry**: `attachments.purge_expired()` deletes attachments older than `ATTACHMENT_TTL_DAYS` (30); a daily background task (`_attachment_gc_loop` in `main.py`'s lifespan) runs it. The Composer shows a dismissible inline notice about the 30-day deletion (`localStorage` `cue-hide-attach-notice`).

**Auth (Google OAuth, Authorization Code flow)** тАФ `routers/auth.py`:
- `GET /auth/google/login` тЖТ signs a `state`, sets it as a `SameSite=Lax` cookie (must survive Google's top-level redirect back), 302s to Google's consent screen.
- `GET /auth/google/callback` тЖТ validates `state` vs the cookie, exchanges the `code` for an access token (`oauth2.googleapis.com/token`, with the client secret), fetches the profile (`openidconnect.googleapis.com/v1/userinfo`). Both calls are stdlib `urllib` over TLS, so no id-token signature check and **no extra runtime dependency**. Rejects unverified emails and anyone failing the allowlist. Upserts the `User` by `google_sub`, issues the session, and тАФ for `OWNER_EMAIL`'s first login тАФ claims all `user_id IS NULL` rows (the original single-user data).
- `GET /auth/me` тЖТ `{authenticated, csrf_token, user}`. `POST /auth/logout` clears cookies.
- The session cookie stays `SameSite=Strict`; only the transient `state` cookie is `Lax`. Session salt is `v2` (carries `uid`), so old single-user sessions are invalidated.

- `lib/api.ts` тАФ typed fetch client; reads `cue_csrf` cookie and sends `X-CSRF-Token` on mutations.
- `lib/markdown.ts` тАФ tiny escape-first Markdown renderer for previews (no DOMPurify needed: HTML is escaped before the markdown subset is applied).

The Composer feeds it a pool of tags already used across prompts (via `usePrompts`) first, then `DEV_TAGS`, deduped case-insensitively. The suggestion list **flips upward** (`.tag-suggest.up`, decided from the input's viewport rect in `recalcDirection`) when there isn't room below тАФ the tag field is the dialog's last field, so a downward menu would otherwise cover the "Anlegen"/"Abbrechen" action buttons. ## Conventions & gotchas

- **No horizontal scroll / mobile**: the board grid uses `repeat(3, minmax(0, 1fr))` (plain `1fr` columns refuse to shrink below a long unbreakable token and blow out the page width); `.column` has `min-width: 0`; card title/preview use `overflow-wrap: anywhere`; `html, body { overflow-x: hidden }` is a final guard. Under 640px the topbar collapses to icon-only tabs (`.tab-label` hidden, brand logo only) so it never exceeds the viewport, and the `.select-bar` wraps.
- **Dialog scroll**: `DetailSheet` (`.sheet--detail`) and `Composer` (`.sheet--composer`) both clip the sheet (`overflow: hidden`) and scroll only their inner wrapper (`.detail-scroll` / `.composer-scroll`, `flex: 1 1 auto; min-height: 0; overflow-y: auto`) while the header and action footer stay fixed (`.sheet--X > * { flex: none }`). Composer additionally pins its fields' height (`.composer-scroll > * { flex-shrink: 0 }`) so the area scrolls instead of squishing. MergeDialog still uses the plain `.sheet` (whole-dialog scroll). **When editing** a prompt the Composer scrim does **not** close on an outside click (`onClick={isEdit ? undefined : onClose}`) so an accidental backdrop click can't discard edits тАФ close via тЬХ/Abbrechen/Esc; a new-prompt Composer still closes on click-out (its draft autosaves anyway).
- **Status transitions**: entering `running`/`done` the first time stamps `ran_at` server-side (in both `PATCH /prompts/{id}` and `/prompts/reorder`). Reorder runs in one transaction.
- **Title derivation**: empty title тЖТ derived from the first non-blank body line (leading `#` stripped), server-side in `_derive_title`.
- **Delete with undo**: prompt deletion (single from the detail, or bulk from the select-bar) goes through `requestDelete(ids)` in `App.tsx` тАФ the rows are hidden immediately (`pendingDelete` filters them out of `filtered`) and the real `DELETE` only fires after a 6 s undo window; the toast's "R├╝ckg├дngig" action cancels it. There is no confirm dialog for prompts (the undo is the safety net); `Confirm` is still used for **project** deletion in `ProjectsView`. Toasts support an optional `action` (`toast.show(msg, tone, { action })`).
- **Merge**: `POST /prompts/merge {source_ids, title, body, project_id, status, tags, originals}` creates the merged prompt and deletes/archives/keeps the sources in one commit (the client composes body/order/format). Requires тЙе2 owned sources.
- **Project order**: `Project.sort_order` is drag-sortable in `ProjectsView` (dnd-kit, `POST /projects/reorder`, optimistic `useReorderProjects`); `list_projects` orders by it, so the filter chips in `App` follow the same order automatically. The chips themselves are also drag-sortable in place (`ProjectChips.tsx`, dnd-kit `rectSortingStrategy`; a `justDragged` ref suppresses the drop-click so finishing a drag doesn't toggle the filter). New projects append (`max(sort_order)+1`).
- **Tenant scoping**: every data router depends on `current_user_id` and filters all reads + re-checks ownership on writes (404 if a row isn't the caller's). New rows always set `user_id`. When adding endpoints, never trust a client-supplied id without an ownership check.
- **Google OAuth secrets**: `GOOGLE_CLIENT_SECRET` (+ `GOOGLE_CLIENT_ID`) live only in `/opt/cue/.env`, never committed, never sent to the browser. The frontend just links to `/api/auth/google/login` (server builds the Google URL). The Google console needs origin `https://cue.celox.io` and redirect URI `https://cue.celox.io/api/auth/google/callback`.
- **Allowlist**: `GOOGLE_ALLOWED_EMAILS` / `GOOGLE_ALLOWED_DOMAINS` (comma lists). Both empty тЖТ nobody can sign in in prod (open access only under `CUE_DEV=1`).
- **CSRF secret lives inside the signed session token** тАФ there is no separate server-side store; the readable `cue_csrf` cookie just mirrors it for the double-submit header.
- **Security headers + CSP** are set in middleware in `main.py`; HSTS is the proxy's job. CSP allows Google Fonts + inline styles (needed for runtime dynamic-color vars).
- **JSX + German quotes**: a straight `"` inside a `"..."` JSX attribute terminates it тАФ use `{'тАжтАЮтАж"тАж'}` (a JS string expression) for German-quoted attribute text.
- **tsconfig**: `tsconfig.node.json` is a composite referenced project and must NOT set `noEmit` (breaks `tsc -b`).
- **Project badge menu (detail sheet)**: the project badge in `DetailSheet` (a `.badge-btn`; prompts without a project get a neutral "Kein Projekt" badge) opens a `.proj-menu` popover with two modes тАФ **Verschieben** (`PATCH /prompts/{id}` via `onMoveProject`) and **Kopieren** (`POST /prompts/{id}/duplicate {project_id}` via `useDuplicatePrompt`/`onCopyToProject`). The duplicate endpoint clones title/body/tags, always lands as **queued**, and duplicates attachment files on disk (`clone_attachment_file` in `attachments.py` тАФ file-path logic stays in that module because its module-level `_settings` is the single source for the attachments dir; calling `get_settings()` elsewhere diverges under the test fixture's cache-clearing). Menu closes on outside click; Escape closes only the menu (capture-phase keydown listener stops propagation so the sheet stays open).
- **Copy interactions**: every prompt is one-click copyable (mini copy button on cards/list rows тАФ styled `.copy-btn`, filled primary like the detail's big copy button; plus the big button in the detail) and **double-click** on a board card or list row copies it. A single click opens the detail, so a 200 ms timer in `PromptCard`/`ListView`'s `ListRow` discriminates click-vs-double-click (double click cancels the pending open тАФ no detail flash).
- **Double-click to edit**: double-clicking the content wrapper in `DetailSheet` (rendered preview or raw `<pre>` тАФ the `contentRef` div; attachments/meta sit outside it) calls `onEdit` and opens the edit composer; double-clicking the composer's own Markdown preview flips `preview` off (the `[preview]` focus effect then focuses the textarea). Trade-off: word-selection via double-click is gone in these areas тАФ select by drag or `Cmd/Ctrl+A`.
- **Composer save shortcut**: `Cmd/Ctrl+Enter` is a **window-level** keydown listener (calls through a render-refreshed `saveRef`, so no stale state), NOT a keydown handler on the sheet element тАФ clicking a non-focusable area (e.g. the preview) moves focus to `<body>`, where an element-bound handler never fires (that was the "Cmd+Enter speichert nicht mehr" bug fixed in 0.7.0). Don't re-add a sheet-level handler alongside it: both firing on one keypress would double-save (duplicate create).
- **Scoped select-all in dialogs**: `DetailSheet`/`Composer`-preview install a `window` keydown listener so `Cmd/Ctrl+A` selects **only** the prompt content (`Range.selectNodeContents` on a ref'd wrapper), not the whole page behind the scrim; `Cmd/Ctrl+C` then copies just that. In the detail, a bare `Cmd/Ctrl+C` with no active selection copies the full prompt. The composer **edit** textarea keeps native select-all (the listener only attaches in preview). The global shortcut handler in `App.tsx` early-returns on any meta/ctrl/alt combo, so these never collide.
- **Last-used project**: creating a prompt stores its project in `localStorage` (`cue-last-project`); the next new prompt preselects it (validated against the live project list; an active project filter still wins via `defaultProjectId`).
- **kbd-in-button legibility**: `.btn kbd` derives its chip background/border from `currentColor` (`color-mix`) so the `тМШтЖ╡` hint stays readable on filled/tonal/danger surfaces instead of dark-on-dark.
- **Conventional Commits**. Persist app state lives entirely in the SQLite file (`/data/cue.db` in the container volume).
