# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`cue` — a multi-tenant prompt-/todo-queue web app for managing Claude-Code-CLI prompts.
Capture prompts, group by project, work them through a status workflow (Queued →
Running → Done, plus Failed/Archived), and copy them into the CLI with one click.
Deployed at `https://cue.celox.io` behind a reverse proxy. Repo: `pepperonas/cue` (private).

Hard constraints: **multi-tenant** — sign in with **Google OAuth**, each user owns their own
projects/prompts (`User` table + `user_id` FK on every owned row), access gated by an
email/domain allowlist. MD3 Expressive design with real spring motion, footer
`© 2026 Martin Pfeffer | celox.io` on every page.

## Commands

```bash
# Backend (from backend/)
uv venv && uv pip install -e ".[dev]"     # setup
uv run uvicorn app.main:app --reload --port 8000   # dev server (set CUE_DEV=1 COOKIE_SECURE=false)
uv run pytest                              # run all tests
uv run pytest tests/test_api.py::test_full_flow    # single test

# Frontend (from frontend/)
pnpm install
pnpm dev                                   # Vite dev server :5173, proxies /api -> :8000
pnpm build                                 # tsc -b && vite build -> dist/
pnpm typecheck
pnpm vitest run                            # frontend unit tests (src/lib only)

# All suites + badge refresh (from the repo root; root package.json)
npm test                                   # backend + runner + frontend, then posttest
npm run update-badges                      # refresh README LOC/test badges (idempotent)

# Full app (prod): one container serves API + built frontend
docker compose up -d --build               # listens on 127.0.0.1:8791
# Auth needs GOOGLE_CLIENT_ID/SECRET + an allowlist in .env (see .env.example).
```

After backend or frontend changes, run `uv run pytest` and `pnpm build` before committing
(or `npm test` from the root to run all three suites — it also refreshes the README badges
via the `posttest` hook).

**Testing conventions:** three suites — `backend/tests/` (pytest, black-box over the HTTP
API with a real tmp SQLite; shared fixtures/helpers in `conftest.py`: `client`, `auth()`,
`make_user()`, `RUNNER_HDR`/`CAPTURE_HDR`), `cue-runner/tests/` (pytest + pytest-asyncio,
fake subprocesses + httpx `MockTransport` + a programmable `FakeApi` for the orchestration
loops in `test_orchestration.py`, no network), `frontend/src/lib/*.test.ts`
(Vitest, only pure lib modules — UI components are deliberately untested). Test behavior,
not implementation; external deps (Google OAuth via monkeypatched `_post_form`/`_get_json`,
subprocesses, browser APIs) are mocked; everything runs offline and deterministically.
Coverage: backend 99 %, runner 91 % (`--cov=app` / `--cov=cue_runner`); remaining gaps are
defensive except-branches, `__main__.py` and a few executor cancel branches. `scripts/update-badges.mjs` parses test counts from `pytest --collect-only` +
`vitest list` — never grep for `it()`.

## Architecture

**One process, one port.** FastAPI mounts the JSON API as a sub-app at `/api` and serves
the built frontend (`STATIC_DIR`, default `static/`) as a SPA for all other paths, with an
`index.html` fallback so client-side state survives hard reloads. In dev the two run
separately and Vite proxies `/api` to `:8000`.

**Backend (`backend/app/`)**
- `main.py` — app assembly, lifespan (`init_db`), security-header + CSP middleware, `/api` mount, SPA static serving (with path-traversal guard).
- `config.py` — env-driven `Settings` (cached). `validate()` fails fast (needs `SECRET_KEY` + `GOOGLE_CLIENT_ID`/`SECRET`) unless `CUE_DEV=1`. `is_email_allowed()` enforces the allowlist (empty lists = closed in prod, open in dev). `google_redirect_uri` derives from `ALLOWED_ORIGIN`.
- `db.py` — SQLite engine; a `connect` event sets `WAL`, `foreign_keys=ON`, `busy_timeout`. `init_db` runs `_migrate()` — idempotent `ALTER TABLE` (PRAGMA-guarded) for additive columns (`bookmarked`, `bookmark_order`, and `user_id` on `prompt`/`project`), since `create_all` only adds missing *tables*, never columns (no Alembic).
- `models.py` — `User` + `Project` + `Prompt` (SQLModel tables). `PromptStatus` enum. Every owned row has a `user_id` FK to `user`; `Prompt.bookmarked`/`bookmark_order` back the bookmarks section; `Prompt.tested` flags whether a running/done prompt's feature was tested. Project name uniqueness is **per user**, not global.
- `schemas.py` — Pydantic request/response models (kept separate from table models). `MeResponse` carries `UserRead`.
- `security.py` — itsdangerous signers: session tokens (payload carries `uid` + the CSRF secret), CSRF double-submit check, and short-lived OAuth-`state` tokens (`issue_oauth_state`/`oauth_state_valid`) to guard the Google redirect. No passwords.
- `deps.py` — `current_session` (401 guard), `current_user_id` (extracts `uid` from the session — inject this into every data router to scope by tenant), `require_csrf` (403 guard + Origin check), `get_client_ip`.
- `routers/` — `auth` (Google OAuth), `projects`, `prompts`, `attachments`, `runs`, `importexport`. **Every data query is filtered by `user_id`; ownership is re-checked on get/update/delete (404 if not owned).**
- **Prompt capture** (`models.CaptureSession/CapturedPrompt`, `routers/capture.py`): every prompt typed in the Claude Code CLI is logged into cue via a `UserPromptSubmit` hook → spool → the cue-runner's forwarder → `POST /api/capture` (Bearer `CAPTURE_TOKEN`, attributed to `OWNER_EMAIL`'s user). Items carry `{session_id, cwd, git_root, prompt, seq}`; the project is derived under `CAPTURE_BASE` (`Settings.capture_project_name`, auto-creates the project): preferred from the hook-reported **git root** relative to the base with `_`-prefixed grouping folders skipped (`_customers/celox/website` → project `celox/website`; an all-underscore root keeps its own name), fallback = first non-`_` `cwd` segment (old hook / no repo / root outside base; `_customers/celox` → `celox`), grouped into a `CaptureSession` per Claude `session_id`, deduped on `(session, seq)`. Owner-facing history: `GET /api/sessions(/{id})` (session detail lists prompts newest-first, `seq` desc), `POST /api/sessions/{id}/prompts/{cp}/promote` (captured → real queued prompt), `DELETE /api/sessions/{id}`. **Multi-tenant:** `/api/capture` resolves the Bearer token to a user — the env `CAPTURE_TOKEN` → `OWNER_EMAIL`, or a per-user `User.capture_token` → that user (with `User.project_base` for derivation). Sessions endpoints + the **Verlauf** tab are per-user (`current_user_id`, not owner-gated); `Settings → Prompt-Capture` lets any user set their base + (re)generate their token. The runner sets `CUE_NO_CAPTURE=1` on its own `claude` execs; its forwarder throttles retry errors. Hook lives in `~/.claude/settings.json` (`python3 …/cue-runner/hooks/capture_hook.py`); runner needs `CAPTURE_TOKEN` in `cue-runner/.env`. All phases done.
- **Send a prompt into a live CLI session** (`models.CliDelivery`, `routers/capture.py`, reverse of capture): the capture hook additionally records each session's terminal context (`ITERM_SESSION_ID`, `TMUX`/`TMUX_PANE` → `CaptureSession.term_program/iterm_session_id/tmux_pane/tmux_socket`, refreshed on every ingest); a session is `deliverable` when a transport is known. Owner-only `POST /api/sessions/{id}/send {text, submit}` enqueues a `CliDelivery` (`queued`); the runner claims it atomically via `GET /api/cli/claim` (guarded by `RUNNER_TOKEN`, mark `sending`), performs it, and reports via `POST /api/cli/{id}/result`; the frontend polls `GET /api/cli/{id}`. **Owner-only** (`require_owner`) because it drives a terminal on the runner's machine, exactly like runs. Runner transport = `cue_runner/deliver.py`: **iTerm2** via `osascript` (AppleScript `write text`, GUID from `ITERM_SESSION_ID`) and **tmux** via `paste-buffer`, both using **bracketed paste** (ESC[200~…ESC[201~) so multi-line prompts land as literal input; then an optional Enter to submit. All ids validated + argv-only (no shell/AppleScript injection). Runner delivery loop polls `GET /api/cli/claim` every `DELIVER_INTERVAL` (1.5 s; `CUE_DELIVER=0` disables). **iTerm2 needs a one-time TCC Automation grant** for the runner's process. Frontend: `SendToSessionDialog` + "An CLI senden" in `DetailSheet` (gated on `canRun`).
- **Run engine** (`models.Run/RunStep/RunLog`, `routers/runs.py`): execute saved prompts through the Claude Code CLI on a Mac **runner** that polls cue. A `Run` is `single` (one step) or `chain` (ordered steps in one Claude session); `RunStep` snapshots the prompt text (reproducible). **User endpoints** (`POST/GET /runs`, `GET /runs/{id}?after_seq=`, `POST /runs/{id}/cancel`, `GET /runs/config`) use cookie auth and are **owner-only** (`require_owner` → `OWNER_EMAIL`; the run executes code on the runner's machine, so it must not be open to other allowlisted users). **Runner endpoints** (`POST /runs/claim`, `/runs/{id}/heartbeat|log`, `/runs/{id}/steps/{idx}/result`, `/runs/{id}/result`) are guarded only by `RUNNER_TOKEN` (`require_runner`, Bearer). Claim is **atomic** (single `UPDATE ... WHERE status='queued' ... RETURNING id`). `project_path` is whitelisted via `Settings.is_path_allowed()` against `ALLOWED_PROJECT_BASES` (string-only on the VPS; the runner re-validates). Chains thread one Claude session; the runner pre-assigns it via `--session-id <uuid>` then `--resume` (CLI v2.1.195 has **no `--cwd`** → set subprocess cwd; **no `--max-turns`** → rely on per-step timeout + `--permission-mode`). Cancel sets `cancel_requested` (running) or `canceled` (queued); the runner learns it from the heartbeat response. The Mac daemon lives in **`cue-runner/`** (own venv/pyproject, PM2-able): asyncio claim-loop, step exec via `create_subprocess_exec` (sets `cwd`, `stdin=DEVNULL` to skip the CLI's stdin wait, parses `stream-json`), heartbeat (also picks up cancel), SIGTERM→SIGKILL cancel, per-step timeout; its own `ALLOWED_BASES` re-validation. `cue-runner/` is **not** deployed to the VPS — it runs on the Mac where `claude` is authenticated. **Frontend** (`RunsView`, `RunDialog`, `state/queries` `useRunConfig/useRuns/useRun/useCreateRun/useCancelRun`): the **Runs** tab + the prompt "Ausführen" action + select-bar "Ausführen" (1 prompt = single, ≥2 = chain) are gated on `useRunConfig` succeeding (owner-only; a 403 hides them, `canRun`). `RunDialog` picks a project base from the whitelist + optional subfolder, model/permission-mode/allowed-tools/bare/skip-permissions/stop-on-error — the last-used settings persist in `localStorage` `cue-run-prefs` and are restored on open (validated against the config whitelists; the **subfolder** deliberately always starts empty); `RunsView` polls (`refetchInterval`) for a live status + log tail, copyable session id, cost, cancel, and re-run. **Hardening (P4):** `runs.reap_stale()` fails runs stuck in claiming/running whose `last_heartbeat` is older than `RUN_STALE_TIMEOUT` (also cleans up runs left in-flight across a backend restart); a `_run_reaper_loop` in the lifespan runs it at startup and every 60 s. The runner heart-beats independently of step progress, so long steps aren't falsely reaped. (P1 backend + P2 runner + P3 frontend + P4 hardening done; P5 = e2e/docs/version + live runner.)
- **Attachments (screenshots)**: `Attachment` table (`user_id`, nullable `prompt_id`, stored `filename`, `original_name`, `content_type`, `size`). Files live in `ATTACHMENTS_DIR` (set to `/data/attachments` in prod so they persist on the volume). `routers/attachments.py`: `POST /attachments` (multipart, image-only, ≤`MAX_ATTACHMENT_BYTES`) returns `{id,url,name,...}` with `prompt_id=null` (pending); `GET /attachments/{id}` serves the file (ownership-checked, not static); `DELETE /attachments/{id}`. Prompts associate them via `attachment_ids` on create/update (`_attach`), and `PromptRead.attachments` is populated by `_reads()`/`_read()` in `prompts.py` (batch-loaded). `delete_prompt` purges files; `merge` reassigns source screenshots to the merged prompt when originals are deleted. **Auto-expiry**: `attachments.purge_expired()` deletes attachments older than `ATTACHMENT_TTL_DAYS` (30); a daily background task (`_attachment_gc_loop` in `main.py`'s lifespan) runs it. The Composer shows a dismissible inline notice about the 30-day deletion (`localStorage` `cue-hide-attach-notice`).

**Auth (Google OAuth, Authorization Code flow)** — `routers/auth.py`:
- `GET /auth/google/login` → signs a `state`, sets it as a `SameSite=Lax` cookie (must survive Google's top-level redirect back), 302s to Google's consent screen.
- `GET /auth/google/callback` → validates `state` vs the cookie, exchanges the `code` for an access token (`oauth2.googleapis.com/token`, with the client secret), fetches the profile (`openidconnect.googleapis.com/v1/userinfo`). Both calls are stdlib `urllib` over TLS, so no id-token signature check and **no extra runtime dependency**. Rejects unverified emails and anyone failing the allowlist. Upserts the `User` by `google_sub`, issues the session, and — for `OWNER_EMAIL`'s first login — claims all `user_id IS NULL` rows (the original single-user data).
- `GET /auth/me` → `{authenticated, csrf_token, user}`. `POST /auth/logout` clears cookies.
- The session cookie stays `SameSite=Strict`; only the transient `state` cookie is `Lax`. Session salt is `v2` (carries `uid`), so old single-user sessions are invalidated.

**Frontend (`frontend/src/`)**
- `lib/color.ts` — Material-You tonal-palette generator from a seed hex (HSL-tone approximation, no heavy dep); `buildSchemes` → light/dark role tokens applied as `--md-*` CSS vars.
- `lib/motion.ts` — MD3 spring presets; all motion respects `prefers-reduced-motion`.
- `lib/api.ts` — typed fetch client; reads `cue_csrf` cookie and sends `X-CSRF-Token` on mutations.
- `lib/markdown.ts` — tiny escape-first Markdown renderer for previews (no DOMPurify needed: HTML is escaped before the markdown subset is applied).
- `lib/tags.ts` — curated English software-dev tag list (`DEV_TAGS`) backing the tag autocomplete.
- `components/TagInput.tsx` — comma-separated tag field with type-ahead. Completion targets the token after the last comma; `↑/↓` navigate, `Enter`/`Tab`/click commit, `Esc` closes. The Composer feeds it a pool of tags already used across prompts (via `usePrompts`) first, then `DEV_TAGS`, deduped case-insensitively. The suggestion list **flips upward** (`.tag-suggest.up`, decided from the input's viewport rect in `recalcDirection`) when there isn't room below — the tag field is the dialog's last field, so a downward menu would otherwise cover the "Anlegen"/"Abbrechen" action buttons. (A sticky action footer was tried and rejected: pinning it to the sheet bottom made it overlap the focused tag input.) **Tags are deduplicated case-insensitively per prompt**: `commit()` won't add a tag the prompt already has, and `normalizeTags`/`dedupeTags` (`lib/tags.ts`) dedup on Composer save + on `PromptCard` render, so a tag can never appear twice on one prompt (kept as a shared helper, first spelling wins).
- `state/queries.ts` — React Query hooks with optimistic updates + rollback (reorder/update/delete).
- `state/settings.tsx` — theme (light/dark/system), seed color, behavior prefs; persisted to localStorage, applied as CSS vars. **Theme switch = circular reveal** (celox.io pattern): `setTheme(mode, origin?)` wraps the change in `document.startViewTransition` and animates a `clip-path: circle()` from the click point on `::view-transition-new(root)` (900 ms desktop / 520 ms small+coarse, `cubic-bezier(0.22,0.08,0,1)`); the DOM flip happens synchronously inside the callback (`applyDom` + `flushSync`) because the new snapshot is taken right after it returns — the idempotent theme `useEffect` re-applies the same values afterwards. `html.theme-transition` (CSS at the end of `global.css`) suppresses the default cross-fade during the reveal; guards: no visual change / reduced motion / no API / reveal already running → instant switch. Callers (TopBar toggle, Settings theme chips) pass the click coords, falling back to the button center on keyboard activation (`clientX/Y === 0`).
- `state/toast.tsx` — toast provider.
- `components/Board.tsx` — dnd-kit multi-container board (cross-column status change + reorder). Local container state syncs from server data except while dragging; `onDragEnd` builds a minimal reorder payload for the affected columns.
- `components/Composer.tsx` — also handles **screenshot uploads**: drag-drop onto the sheet, paste (Cmd/Ctrl+V), or the "Bild hinzufügen" button → `api.uploadAttachment` (pending, `prompt_id=null`), thumbnails with remove. On save it passes `attachment_ids`; on cancel/unmount it deletes the still-uncommitted uploads (`savedRef`/`newIds`). `DetailSheet` shows attachment thumbnails (click → `.lightbox`); cards show a count chip. **Voice dictation**: a mic chip next to the prompt label drives `useDictation` (`lib/speech.ts`, browser-native Web Speech API — no backend, own minimal typings since `SpeechRecognition` isn't in lib.dom). Finalized phrases append to the body with smart spacing; interim text shows in `.dictation-live` below the textarea. The hook auto-restarts recognition when Chrome ends it on silence (`wantRef` carries user intent), aborts on unmount, and reports `supported:false` on Firefox (button hidden). Switching to preview stops recording (the chip would be invisible). **Gotcha:** the backend's `Permissions-Policy` header must keep `microphone=(self)` — `microphone=()` hard-blocks `getUserMedia`/speech even for same-origin.
- `components/MergeDialog.tsx` — merge several prompts into one: reorderable source list (↑/↓, remove), format choice (titles-as-headings / `---` rule / blank line), live markdown preview, smart prefills (title defaults to the source titles joined `A [&] B` and follows reordering until edited; union of tags; common project), and what to do with the originals (delete/archive/keep). Confirm → `POST /prompts/merge` (`useMergePrompts`). Selection itself is App state (`selectMode` + ordered `selectedIds`); cards/list rows show a checkbox and toggle selection instead of opening, drag is disabled while selecting, and a fixed `.select-bar` offers "Zusammenführen" (≥2). **Cmd/Ctrl+click** on a card/row toggles selection directly (`modSelect` in App drives `selectMode` from the result: first mod+click enters selection, emptying it leaves — also from explicit chip mode); a mod+dblclick never copies (`handleDoubleClick` early-returns on `metaKey/ctrlKey`). Shift was deliberately dropped as the modifier (0.11.1); note macOS Ctrl+click = system right-click, so Mac users use Cmd. The `.select-bar` is deliberately NOT inside `AnimatePresence` — its exit animation never visibly played (froze ~2 s at full opacity, then popped, spring and tween alike), so the bar unmounts instantly; the spring entrance stays.
- `components/BookmarksView.tsx` — the **Bookmarks** tab: a single drag-sortable column (dnd-kit `SortableContext`, reuses `PromptCard`) of all `bookmarked` prompts ordered by `bookmark_order`. Local order syncs from server except while dragging; `onDragEnd` → `arrayMove` → `POST /prompts/bookmarks/reorder` (optimistic via `useReorderBookmarks`). Bookmarking elsewhere (card/list/detail bookmark toggle → `PATCH {bookmarked}`) appends to the end of this section server-side.
- `components/ListView.tsx` — list view grouped by status into **collapsible** sections (one per visible status, driven by the same `columns` prop as the board, so the "Failed / Archived" toggle applies here too). Collapse state per status persists in `localStorage` (`cue-list-collapsed`); section bodies animate height with `emphasized` (reduced-motion → instant). Status icons are tinted via `STATUS_CLASS` (`.st-queued/.st-running/.st-done/.st-failed/.st-archived` in `global.css`) — the same subtle tint is applied to the board column heads, cards, and detail sheet.
- `App.tsx` — auth gate, filters/search, keyboard shortcuts, view switching, mounts Composer/Detail/Confirm/Shortcuts overlays. The active view (`board/list/bookmarks/projects/settings`) persists to `localStorage` (`cue-view`) and the active project filter to `cue-project-filter` (reset to `all` if that project was since deleted) — both restored on reload.
- `components/ToggleIconButton.tsx` — generic animated icon toggle (tints + fills when active, icon `motion.span` keyed by state so it pops on every flip; reduced-motion → instant). `BookmarkButton` (gold, `bookmark`/`bookmark_border`) and `TestedButton` (green, `verified` with `FILL` axis via `.tested-btn.active`) are thin wrappers. The tested toggle is only rendered for **running/done** prompts (card actions, list rows, detail header), backed by `Prompt.tested`.

## Conventions & gotchas

- **No horizontal scroll / mobile**: the board grid uses `repeat(3, minmax(0, 1fr))` (plain `1fr` columns refuse to shrink below a long unbreakable token and blow out the page width); `.column` has `min-width: 0`; card title/preview use `overflow-wrap: anywhere`; `html, body { overflow-x: hidden }` is a final guard. Under 640px the topbar collapses to icon-only tabs (`.tab-label` hidden, brand logo only) so it never exceeds the viewport, and the `.select-bar` wraps.
- **Dialog scroll**: `DetailSheet` (`.sheet--detail`) and `Composer` (`.sheet--composer`) both clip the sheet (`overflow: hidden`) and scroll only their inner wrapper (`.detail-scroll` / `.composer-scroll`, `flex: 1 1 auto; min-height: 0; overflow-y: auto`) while the header and action footer stay fixed (`.sheet--X > * { flex: none }`). Composer additionally pins its fields' height (`.composer-scroll > * { flex-shrink: 0 }`) so the area scrolls instead of squishing. MergeDialog still uses the plain `.sheet` (whole-dialog scroll). **When editing** a prompt the Composer scrim does **not** close on an outside click (`onClick={isEdit ? undefined : onClose}`) so an accidental backdrop click can't discard edits — close via ✕/Abbrechen/Esc; a new-prompt Composer still closes on click-out (its draft autosaves anyway).
- **Status transitions**: entering `running`/`done` the first time stamps `ran_at` server-side (in both `PATCH /prompts/{id}` and `/prompts/reorder`). Reorder runs in one transaction. **Done goes on top**: a PATCH to `done` uses `_top_sort_order` (min-1) instead of appending; the Board also forces a cross-column drag into done to index 0 (`dragOrigin` ref in `onDragEnd`). A successfully finished run step moves its source prompt to done-top, a failed one to failed (`runs.step_result`).
- **Board mirrors runs** (`runs._prompts_to_running`/`_release_unfinished_prompts`): `create_run` moves all source prompts into Running (append order = step order, `ran_at` stamped, blocked skipped); when a run ends without executing all steps — cancel of a queued run, run result failed/canceled (stop-on-error), or `reap_stale` — prompts of steps not in `RUN_TERMINAL` that still sit in Running return to queued. Frontend: `useCreateRun`/`useCancelRun` invalidate the prompts query so the board updates immediately; run finish updates arrive via the RunTicker's invalidation.
- **Blocked status** (`Prompt.blocked`, migration in `db.py`): toggle left of the bookmark (`BlockedButton`, red tint), **only rendered for `queued` prompts** (`canBlock` in PromptCard/ListRow/DetailSheet, analogous to `canTest`) — blocked only makes sense in the queue. Server-side: `blocked=true` on a non-queued prompt → 400; leaving queued (also via `/prompts/reorder`) clears the flag; an idempotent data fix in `db._migrate()` clears stale flags. Blocked prompts sort to the bottom of their column (client-side sort key `Number(blocked)` in Board `group()`, ListView, `navOrder`), are grayed (`.card.blocked`), not draggable (attributes/listeners omitted), and the server rejects `running`/`done` with 400 while blocked (`update_prompt` guard — applied AFTER `payload.blocked`, so unblock+move works in one PATCH). All client status changes go through `applyStatus` in App (single gate + toast); "Ausführen" is hidden for blocked prompts.
- **Board column cap**: max 10 cards per column (`COLUMN_CAP` in Board), rest behind a per-column "+N weitere anzeigen" expander; while a drag is active ALL cards render (so drops beyond the cap stay possible).
- **Run ticker** (`RunTicker.tsx`): floating pill (fixed bottom-left) while any run is queued/claiming/running, mounted app-wide except on the Runs tab; shares the `useRuns` query with RunsView (polling stops when nothing is active). On an active→terminal transition it invalidates the prompts query (board reflects the run's done/failed move) and toasts the outcome. **Step progress**: `RunRead` carries `steps_done`/`steps_total` (steps in a terminal state vs. all, computed in `runs._run_read`/`_run_detail`) so list views get progress without loading details — the ticker appends „· 4/5 abgeschlossen" when total > 1 (summed across active runs), the Runs tab shows a `4/5` chip (`.run-progress`) in playbook header rows.
- **Runs tab expand state** (`RunsView.tsx`): each run row is a `RunCard` that fetches/polls its own detail via `useRun(open ? id : null)`, so several can be open at once. **Active runs (queued/claiming/running) are expanded by default**; a manual toggle wins via `openOverride` (per-id `Record<string, boolean>`) — an untouched active run therefore auto-collapses when it finishes.
- **Runner concurrency**: `MAX_CONCURRENCY` in `cue-runner/.env` (semaphore in `run_forever`) — currently 3 parallel runs, each its own Claude session.
- **Title derivation**: empty title → derived from the first non-blank body line (leading `#` stripped), server-side in `_derive_title`.
- **Delete with undo**: prompt deletion (single from the detail, or bulk from the select-bar) goes through `requestDelete(ids)` in `App.tsx` — the rows are hidden immediately (`pendingDelete` filters them out of `filtered`) and the real `DELETE` only fires after a 6 s undo window; the toast's "Rückgängig" action cancels it. There is no confirm dialog for prompts (the undo is the safety net); `Confirm` is still used for **project** deletion in `ProjectsView`. Toasts support an optional `action` (`toast.show(msg, tone, { action })`).
- **Merge**: `POST /prompts/merge {source_ids, title, body, project_id, status, tags, originals}` creates the merged prompt and deletes/archives/keeps the sources in one commit (the client composes body/order/format). Requires ≥2 owned sources.
- **Project order**: `Project.sort_order` is drag-sortable in `ProjectsView` (dnd-kit, `POST /projects/reorder`, optimistic `useReorderProjects`); `list_projects` orders by it, so the filter chips in `App` follow the same order automatically. The chips themselves are also drag-sortable in place (`ProjectChips.tsx`, dnd-kit `rectSortingStrategy`; a `justDragged` ref suppresses the drop-click so finishing a drag doesn't toggle the filter). New projects append (`max(sort_order)+1`).
- **Tenant scoping**: every data router depends on `current_user_id` and filters all reads + re-checks ownership on writes (404 if a row isn't the caller's). New rows always set `user_id`. When adding endpoints, never trust a client-supplied id without an ownership check.
- **Google OAuth secrets**: `GOOGLE_CLIENT_SECRET` (+ `GOOGLE_CLIENT_ID`) live only in `/opt/cue/.env`, never committed, never sent to the browser. The frontend just links to `/api/auth/google/login` (server builds the Google URL). The Google console needs origin `https://cue.celox.io` and redirect URI `https://cue.celox.io/api/auth/google/callback`.
- **Allowlist**: `GOOGLE_ALLOWED_EMAILS` / `GOOGLE_ALLOWED_DOMAINS` (comma lists). Both empty → nobody can sign in in prod (open access only under `CUE_DEV=1`).
- **CSRF secret lives inside the signed session token** — there is no separate server-side store; the readable `cue_csrf` cookie just mirrors it for the double-submit header.
- **Security headers + CSP** are set in middleware in `main.py`; HSTS is the proxy's job. CSP allows Google Fonts + inline styles (needed for runtime dynamic-color vars).
- **JSX + German quotes**: a straight `"` inside a `"..."` JSX attribute terminates it — use `{'…„…"…'}` (a JS string expression) for German-quoted attribute text.
- **tsconfig**: `tsconfig.node.json` is a composite referenced project and must NOT set `noEmit` (breaks `tsc -b`).
- **Project badge menu (detail sheet)**: the project badge in `DetailSheet` (a `.badge-btn`; prompts without a project get a neutral "Kein Projekt" badge) opens a `.proj-menu` popover with two modes — **Verschieben** (`PATCH /prompts/{id}` via `onMoveProject`) and **Kopieren** (`POST /prompts/{id}/duplicate {project_id}` via `useDuplicatePrompt`/`onCopyToProject`). The duplicate endpoint clones title/body/tags, always lands as **queued**, and duplicates attachment files on disk (`clone_attachment_file` in `attachments.py` — file-path logic stays in that module because its module-level `_settings` is the single source for the attachments dir; calling `get_settings()` elsewhere diverges under the test fixture's cache-clearing). Menu closes on outside click; Escape closes only the menu (capture-phase keydown listener stops propagation so the sheet stays open).
- **Copy interactions**: every prompt is one-click copyable (mini copy button on cards/list rows — styled `.copy-btn`, filled primary like the detail's big copy button; plus the big button in the detail) and **double-click** on a board card or list row copies it. A single click opens the detail, so a 200 ms timer in `PromptCard`/`ListView`'s `ListRow` discriminates click-vs-double-click (double click cancels the pending open — no detail flash).
- **Double-click to edit**: double-clicking the content wrapper in `DetailSheet` (rendered preview or raw `<pre>` — the `contentRef` div; attachments/meta sit outside it) calls `onEdit` and opens the edit composer; double-clicking the composer's own Markdown preview flips `preview` off (the `[preview]` focus effect then focuses the textarea). Trade-off: word-selection via double-click is gone in these areas — select by drag or `Cmd/Ctrl+A`.
- **Composer save shortcut**: `Cmd/Ctrl+Enter` **and `Cmd/Ctrl+S`** save from anywhere in the dialog — a **capture-phase window** keydown listener (calls through a render-refreshed `handleSaveKeyRef`) plus a bubble-phase backup handler on the sheet. The two coordinate via a **WeakSet of handled events, NOT via `defaultPrevented`**: the old `!e.defaultPrevented` guard meant any earlier listener that preventDefaults the combo (e.g. a browser extension's capture listener) silently disabled saving entirely — that was the "Cmd+Enter does nothing" bug. `Cmd/Ctrl+S` is the muscle-memory fallback for setups where something outside the page swallows Cmd+Enter (preventDefault suppresses the browser save dialog). An **empty body no longer no-ops silently** (error toast "Prompt-Text fehlt" + focus back to the textarea), and a `savingRef` in-flight guard prevents double-creates from rapid repeats (button routes through the same `triggerSave`). Background: an element-bound handler alone missed the combo whenever focus sat on `<body>` (e.g. after clicking the preview).
- **Scoped select-all in dialogs**: `DetailSheet`/`Composer`-preview install a `window` keydown listener so `Cmd/Ctrl+A` selects **only** the prompt content (`Range.selectNodeContents` on a ref'd wrapper), not the whole page behind the scrim; `Cmd/Ctrl+C` then copies just that. In the detail, a bare `Cmd/Ctrl+C` with no active selection copies the full prompt. The composer **edit** textarea keeps native select-all (the listener only attaches in preview). The global shortcut handler in `App.tsx` early-returns on any meta/ctrl/alt combo, so these never collide.
- **Last-used project**: creating a prompt stores its project in `localStorage` (`cue-last-project`); the next new prompt preselects it (validated against the live project list; an active project filter still wins via `defaultProjectId`).
- **Header project label**: the board view shows the active project filter next to the brand (`TopBar` `projectLabel` prop, `.brand-project` chip, remounted `motion.span` per label so the spring replays; hidden <760px).
- **kbd-in-button legibility**: `.btn kbd` derives its chip background/border from `currentColor` (`color-mix`) so the `⌘↵` hint stays readable on filled/tonal/danger surfaces instead of dark-on-dark.
- **Conventional Commits**. Persist app state lives entirely in the SQLite file (`/data/cue.db` in the container volume).
